# CLI / GUI 機能比較（実行可能性ベース）

## 前提
本比較は「ユーザーが今このリポジトリで実行できる機能」を基準にする。

## 1. 実行可否サマリ

| 区分 | エントリ | 実行可否 | 失敗要因 |
|---|---|---|---|
| CLI | `python -m karuku_resizer.resize_core` | 不可 | `resize_core.py` の構文エラー |
| GUI | `python -m karuku_resizer.gui_app` | この環境では不可 | `customtkinter` 依存不足 |

補足：GUIは依存が揃えば起動可能性が高いが、本セッション環境では未確認。

## 2. CLI機能（仕様上 vs 実行可能）

### 2.1 引数定義上の意図
`resize_core.py` には以下のCLI引数が記載されている。
- `-s/--source`
- `-d/--dest`
- `-w/--width`
- `-q/--quality`
- `-f/--format`
- `--dry-run`
- `-v/--verbose`

### 2.2 実行可能性
- 現在は `SyntaxError` で停止するため、実行可能機能は実質なし。

## 3. GUI機能（コード実装ベース）

### 3.1 実装済み
1. 画像選択（複数）
2. リサイズモード（比率/幅/高さ/固定）
3. プレビュー
4. 単体保存
5. 一括保存（進捗・キャンセル）
6. ズーム操作
7. 設定保存

### 3.2 未実装（今回追加候補）
1. 品質指定UI
2. 出力形式指定UI
3. ドライラン
4. 詳細ログ切替

## 4. 「ユーザーが実行できる機能」を厳密に言うと
- 現在のセッション環境で厳密に言えば、CLI/GUIとも即時実行は不可。
- ただし、GUIは依存導入で利用可能になる見込みがある（コード構造上）。

## 5. リスク評価
1. 仕様説明だけで実装判断すると、実際のコードとの差分で手戻りが出る。
2. CLI修復とGUI新機能追加を同時に行うと、デバッグ面で複雑化する。
3. まずGUI経路で要件を満たす設計に集中するのが現実的。

## 6. 推奨運用（この比較に基づく）
1. GUIを主軸として新機能を実装。
2. CLIは別タスクとして構文/参照修復を先行し、その後に機能追従。
3. ドキュメントは実装に合わせて更新。
