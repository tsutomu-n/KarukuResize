# スケジュール案（個人開発前提）

## 前提
- 個人開発、短時間で価値を出すことを優先。
- まずPhase 1を完了してからPhase 2へ。
- 1日あたりの集中作業を 2〜4時間想定。

## 1. 推奨スケジュール（5〜7日）

### Day 1: 設計固定 + UI骨組み
1. GUIに「出力設定」「EXIF設定」セクション追加
2. 設定保存キーを追加
3. 既存UIとのレイアウト整合

成果物:
- UIは表示されるが処理未接続でも可

### Day 2: 保存処理の共通化 + 品質/形式
1. 単体保存/一括保存の内部保存処理を共通関数化
2. 品質（5刻み）反映
3. 出力形式分岐の導入
4. 対応形式の実行時判定（webp/avif）

成果物:
- 形式と品質が保存結果に反映

### Day 3: EXIF keep/remove + GPS削除
1. EXIF抽出/付与処理を追加
2. keep/removeを保存処理へ接続
3. GPS削除処理の追加

成果物:
- EXIF保持/削除/GPS削除が機能

### Day 4: ドライラン + 詳細ログ + 結果表示
1. ドライラン導入（保存スキップ）
2. 詳細ログ切替導入
3. 保存結果ダイアログ（EXIF反映件数等）

成果物:
- ユーザーが安全に試せる状態

### Day 5: 検証 + ドキュメント更新
1. 手動テスト（単体/一括、形式別）
2. 既知不具合修正
3. README/QUICK_START更新

成果物:
- Phase 1完了版

### 予備 Day 6-7（推奨）
1. 仕様微調整
2. UI文言改善
3. 追加の回帰確認

## 2. Phase 2 スケジュール（3〜4日）

### Day A
- edit UI + バリデーション

### Day B
- EXIF編集反映（4項目限定）

### Day C
- 差分プレビュー + エラーハンドリング

### Day D（予備）
- 検証・調整

## 3. マイルストーン
1. M1: 品質/形式/ドライラン/ログ が使える
2. M2: EXIF keep/remove/GPS削除 が使える
3. M3: edit（限定項目）+ 差分プレビュー が使える

## 4. 進行管理チェックリスト
1. 毎日終了時に「完了/未完了/ブロッカー」を1行で記録
2. 仕様変更は必ずドキュメントへ反映
3. 実装とドキュメントの乖離を残さない
